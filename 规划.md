# S-TextPaste 程序规划文档

## 1. 项目定义

**S-TextPaste** 是一个“零信任”端到端加密文本分享服务，专为追求极限安全的用户设计。使用者可以在浏览器内编写 Markdown 文本，所有内容均在本地加密后再上传至服务器，服务端在任何情况下均无法解密用户数据。  
项目以“绝密级”安全为默认基线，启用后量子混合加密、双信封加密，并强制要求设置密码（或自定义密钥）。目标是通过 Cloudflare Workers 实现一键部署、无服务器化运行。

- **仓库地址**：`https://github.com/stop666two/S-TextPaste`
- **部署平台**：Cloudflare Workers + D1 Database
- **核心价值**：即使服务端完全被攻破，用户内容依然安全。

## 2. 核心特性清单

### 2.1 极限加密
- **浏览器端加密**：所有加解密操作均在用户浏览器内执行，利用 Web Crypto API 及后量子库，服务器只收发密文。
- **强制密码保护**：创建粘贴时必须设置密码，无法跳过。密码通过 SHA-256 + MD5 复杂排序算法生成高熵加密密钥。
- **多重密钥体系**：
  - 密码派生密钥（默认推荐）
  - 用户自定义对称密钥（直接提供 AES 密钥）
  - 用户自定义非对称密钥（RSA/EC 公钥加密数据密钥）
- **后量子加密（PQC）**：默认集成 NIST 标准 Kyber-1024 密钥封装机制，确保在量子计算时代仍具备长期安全性。
- **双信封加密（双层保护）**：数据密钥先由后量子 KEM 封装，后量子私钥再由密码派生密钥加密，形成纵深防御。
- **全算法支持**：AES-128/192/256-GCM、RSA-OAEP、ECDH（P-256/P-384/P-521）、Kyber-1024 混合使用。

### 2.2 Markdown 全功能支持
- 基于 CodeMirror 6 的专业编辑器，支持语法高亮、自动补全。
- 实时分屏预览：编辑区与渲染视图同步滚动。
- 扩展语法：GFM（表格、任务列表）、Mermaid 流程图、KaTeX 数学公式、代码块高亮。

### 2.3 高级生命周期控制
- **过期时间**：可设定粘贴在指定时长后自动销毁。
- **阅后即焚**：首次阅读成功后立即从服务器永久删除。
- **最大浏览次数**：超过设定上限后自动失效。
- **密码错误锁定**：连续解密失败 5 次，该粘贴暂时拒绝访问（可配置）。

### 2.4 自定义与便利性
- **自定义短链接**：允许用户设置个性化 URL 后缀。
- **密码提示**：可填写提示信息，协助记忆（提示本身不参与加密）。
- **暗黑模式**：界面自适应系统主题。
- **多语言**：中文/英文界面切换。
- **安全性仪表盘**：解密页面向用户展示当前粘贴的加密算法、密钥强度、量子安全等级等信息。

### 2.5 一键部署与零运维
- 提供完整 `wrangler.toml` 配置，一条命令部署到 Cloudflare Workers。
- 使用 Cloudflare D1（SQLite 兼容）作为持久化存储，自动建表。
- 内置 API 文档与安全策略头。

## 4. 系统架构

```
┌───────────────┐          HTTPS           ┌──────────────────────┐
│   浏览器       │ ◄──────────────────────► │  Cloudflare Workers  │
│  - 加密/解密   │    密文 + 元数据          │  (API 路由)          │
│  - 编辑器/预览 │                          │  - 粘贴 CRUD         │
│               │                          │  - 访问控制          │
└───────────────┘                          └──────────┬───────────┘
                                                      │
                                                Cloudflare D1
```

- 浏览器承担全部加密逻辑：生成数据密钥、执行密钥封装、构造安全信封。
- Workers 仅提供 RESTful API 接口，负责读写数据库，不做任何加解密运算。
- D1 存储完全不可读的密文及少量元数据，无明文风险。

## 5. 详细加密方案

### 5.1 密码派生密钥（复杂排序算法）

当用户选择密码模式时，系统生成 32 字节随机盐 `S`，结合用户输入的密码 `P` 进行以下派生：

1. 计算 `H1 = SHA-256(P || S)` （32 字节）
2. 计算 `H2 = MD5(P || S)` （16 字节）
3. 将两个哈希值按字节交错排列：  
   `K_raw = H1[0] | H2[0] | H1[1] | H2[1] | ... | H1[15] | H2[15] | H1[16] | ... | H1[31]`  
   （前 16 对交错，后 16 字节直接取 SHA-256 剩余部分）
4. 取 `K_raw` 的前 32 字节作为最终 AES-256 密钥。

该混合方式显著提高对抗彩虹表和定制硬件攻击的能力，且由于盐值随机，相同密码每次生成不同密钥。

### 5.2 双信封+后量子绝密模式（默认）

当用户使用密码并开启后量子加密（默认启用）时，完整流程如下：

**加密过程：**
1. 生成随机数据对称密钥 `DK`（32字节），使用 AES-256-GCM 加密 Markdown 文本，得到 `ciphertext`。
2. 生成一次性 Kyber-1024 密钥对 `(pq_pk, pq_sk)`。
3. 使用 `pq_pk` 进行 Kyber 封装，得到 `(ct_pq, ss_pq)`，再利用 HKDF 从 `ss_pq` 衍生出 `DK`（实际为：直接使用 `ss_pq` 作为派生密钥的材料，输出 32 字节密钥）。**此处修正为**：我们不再用 `DK` 直接加密，而是用 `ss_pq` 作为密钥加密文本，从而 `ct_pq` 是封装结果，解密时需 `pq_sk` 恢复 `ss_pq`。但为了保持双信封的清晰性，我们采用如下结构：
   - 随机生成 `DEK`（数据加密密钥），用 `DEK` 加密明文得到 `encrypted_text`。
   - 使用 Kyber 封装 `DEK`：`(ct_pq, ss_pq) = Kyber.Encaps(pq_pk)`，然后用 `ss_pq` 作为密钥材料，使用 HKDF 派生一个密钥 `KEK`，再用 `KEK` 加密 `DEK` 得到 `encrypted_dek`。（这样更标准）
   - 实际上可简化为：直接用 `ss_pq` 作为 AES-GCM 密钥加密明文。为统一，我们采用：用 Kyber 封装的共享秘密作为根密钥，直接加密数据。
4. 使用密码派生密钥 `AES_Key`（来自 5.1）加密 `pq_sk`，得到 `enc_pq_sk`。
5. 最终信封内容：`encrypted_text`、`ct_pq`、`enc_pq_sk`、盐 `S`、密码提示（可选）。

**解密过程：**
1. 用户输入密码，结合盐 `S` 派生 `AES_Key`。
2. 用 `AES_Key` 解密 `enc_pq_sk` 获得 `pq_sk`。
3. 使用 `pq_sk` 对 `ct_pq` 解封装，得到 `ss_pq`。
4. 用 `ss_pq`（或派生密钥）解密 `encrypted_text`，还原明文 Markdown。

这种双重保护意味着，即使未来的量子计算机能够破解密码派生算法，攻击者仍需先攻破 Kyber 密文，而 Kyber 是抗量子的；同时，即使 Kyber 出现弱点，攻击者也无法获取受密码保护的后量子私钥。

### 5.3 自定义对称密钥模式

用户可以自行输入一个 Base64 编码的对称密钥。客户端处理：
- 若长度恰好为 32 字节，直接作为 AES-256 密钥。
- 否则，使用 HKDF-SHA-256（盐为空，info 为 `"S-TextPaste-symmetric"`）派生固定 32 字节密钥。
- 用该密钥直接加密明文，生成 `encrypted_text`，信封不包含 `pq_sk` 或其它后量子材料，仅标记 `mode: "symmetric"`。

服务端无需存储盐值，但需要能够区分模式。

### 5.4 自定义非对称密钥模式

支持用户上传或粘贴公钥（PEM 或 JWK 格式），支持 RSA-OAEP、EC P-256/384/521。加密流程：
1. 生成随机数据加密密钥 `DEK`（32 字节）。
2. 用 `DEK`（AES-256-GCM）加密明文。
3. 使用用户公钥加密 `DEK`，得到 `encrypted_dek`。
4. 信封存储：`encrypted_text`、`encrypted_dek`、算法标识、公钥指纹。
解密时，用户需提供对应私钥（浏览器本地导入，不上传服务器），客户端使用私钥解密 `encrypted_dek` 得到 `DEK`，再解密密文。

### 5.5 数据完整性与认证

所有 AES-GCM 加密均自带认证标签，客户端在解密前验证标签，防止篡改。信封内 JSON 结构包含版本号与算法标识，以支持未来扩展。

## 6. API 接口设计

Base URL: `https://<your-worker>.workers.dev/api`

| 方法   | 路径                | 描述                   | 请求体/参数                                                                 | 返回                                                                 |
|--------|---------------------|------------------------|-----------------------------------------------------------------------------|----------------------------------------------------------------------|
| POST   | `/paste`            | 创建加密粘贴           | JSON: `{ mode, salt?, encrypted_payload, hint?, expires_in?, max_views?, burn?, custom_id? }` | `{ id, delete_token, expires_at }`                                  |
| GET    | `/paste/:id`        | 获取粘贴密文及元数据   | 无                                                                          | `{ mode, salt?, encrypted_payload, hint?, expires_at, view_count, burn, ... }` |
| DELETE | `/paste/:id`        | 删除粘贴               | Header: `X-Delete-Token`（创建时返回的 token）                             | `{ success: true }`                                                  |
| POST   | `/paste/:id/view`   | 记录一次访问           | 无（前端在每次加载时调用，用于触发计数和阅后即焚逻辑）                     | `{ success: true }`                                                  |

- `delete_token` 由服务器生成，为 256 位随机值，数据库仅存储其 SHA-256 哈希以验证删除请求。
- 所有响应均包含严格的 CSP 头，防止 XSS。

## 7. 数据库设计（Cloudflare D1）

```sql
CREATE TABLE IF NOT EXISTS pastes (
  id TEXT PRIMARY KEY,
  mode TEXT NOT NULL,                -- 'password', 'symmetric', 'asymmetric'
  salt TEXT,                         -- Base64 salt, 仅密码模式
  encrypted_payload TEXT NOT NULL,   -- 整个加密信封的 Base64
  hint TEXT DEFAULT '',
  delete_token_hash TEXT NOT NULL,
  expires_at INTEGER,                -- Unix 时间戳，NULL 表示永不过期
  max_views INTEGER DEFAULT -1,      -- -1 表示无限
  view_count INTEGER DEFAULT 0,
  burn_after_read INTEGER DEFAULT 0, -- 0 或 1
  created_at INTEGER NOT NULL,
  pubkey_fingerprint TEXT            -- 非对称模式下公钥指纹，便于用户识别
);
```

- 主键 `id` 为用户自定义短链接或随机生成的 8 位 nanoid。
- 所有敏感数据均位于 `encrypted_payload` 中，服务器无法解析。

## 8. 前端界面与交互

### 8.1 创建页面
- **编辑器区**：占据左侧（或上方），提供 Markdown 编写，实时预览在右侧（或下方）。
- **加密配置面板**：
  - 加密模式选择：密码 / 对称密钥 / 非对称公钥。
  - 若选密码：输入密码 + 确认密码，可选密码提示。
  - 后量子开关（默认开启，仅密码模式下可见，因其他模式由用户自行决定）。
  - 生命周期：过期时间选择器、最大浏览次数、阅后即焚复选框。
  - 自定义 URL 后缀输入框。
- 点击“创建加密粘贴”，客户端完成所有加密工作，上传并生成分享链接，展示给用户（附带删除令牌）。

### 8.2 阅读页面
- 无任何内容可见，仅显示一个解密表单。
- 根据 `mode` 动态显示输入项：
  - 密码模式：密码框 + 提示（若有）。
  - 对称模式：密钥文本域。
  - 非对称模式：私钥文件选择或粘贴区域。
- 点击“解密”，浏览器内执行解密，成功后跳转至 Markdown 渲染视图。
- 解密失败时给出明确提示，若连续失败 5 次，显示锁定信息。
- 渲染视图中包含**安全性仪表盘**：显示当前加密算法、密钥强度、量子安全状态。

### 8.3 其他
- 界面支持中英文切换。
- 自动检测系统暗色模式并应用。
- 所有静态资源开启子资源完整性（SRI）校验。

## 9. 一键部署到 Cloudflare Workers

### 9.1 前置条件
- Node.js 18+ 与 pnpm
- Cloudflare 账号，并安装 Wrangler CLI：`npm install -g wrangler`
- 创建 D1 数据库：
  ```bash
  wrangler d1 create s-textpaste-db
  ```
  记下返回的 `database_id`。

├── worker/                 # Workers 代码

### 9.3 wrangler.toml 配置示例
```toml
name = "s-textpaste"
main = "src/index.ts"
compatibility_date = "2024-12-01"

[[d1_databases]]
binding = "DB"
database_name = "s-textpaste-db"

[site]

```

### 9.4 部署命令
```bash
git clone https://github.com/stop666two/S-TextPaste
cd S-TextPaste
pnpm install
pnpm run deploy
```
`deploy` 脚本将依次执行：
1. 构建前端，输出至 `worker/public/`。
2. 运行 `wrangler deploy`，推送 Worker 及静态站点。

首次部署后，访问分配的 `*.workers.dev` 域名即可使用。D1 表将在 Worker 首次处理请求时自动创建。

## 10. 安全策略与威胁模型

- **零信任假设**：服务端不可信，客户端执行所有密码运算，不信任任何中间网络。
- **无明文记录**：Worker 日志不记录密码、密钥、解密内容或任何敏感字段。
- **认证加密**：统一使用 AES-256-GCM，防止密文篡改。
- **访问令牌保护**：删除令牌仅存储其哈希，即使数据库泄露，攻击者也无法删除粘贴。
- **侧信道缓解**：浏览器环境内，使用 Web Crypto API 的常数时间实现；密码比较使用固定时间比较函数。
- **量子安全**：默认 Kyber-1024 保护，可抵抗未来量子计算机攻击。
- **内容安全策略 (CSP)**：Worker 响应头设置为 `Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline';` 等严格策略。
- **子资源完整性 (SRI)**：前端引用的第三方库（如 liboqs-wasm）将计算并硬编码 SRI 哈希。
